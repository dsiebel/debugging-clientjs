the debuggable application

how can my application help me debug itself?
ideas:
  • understand errors work and their caveats
  • name your anonymous functions
  • message bus system (Wreqr: Commands, RequestResponse, EventAggregator)
  • log everything
  • other tools

understand {object Error}

message, name, description, number, fileName, lineNumber, columnNumber, stack, toSource(), toString()

but be aware

and this is just the beginning ...

Incompatibilities:
  • fileName is not standard
  • lineNumber ist not standard
  • columnNumber only sometimes available
  • IE breaks differently with every version
  • Some browsers put properties on instances other on prototype
  • Format of the stacktrace varies accross browsers
  • Chrome uses line, Firefox uses lineNumber
  • Not all properties will be visible in the debugger (not always treated as object)

error bubbling

the way of the error - async edition
Now this get's even more interesting when we cross
the "asynchronous boundary"
  • function foo()
  • ➥ native code
  • ➥ function bar()

again we have our function foo that calls addEventListener or setTimeout or any other native function that accepts a callback that will be executed at some point in time

when the user then clicks something or the timeout elapses our callback gets executed

  • function foo()
  • ➥ native code
  • ➥ function bar()
and an error happens in our callback
  • function foo()
  • ➥ native code
  • ➦ function bar()
again the error bubbles all the way up to the native code
  • function foo()
  • ➦ native code
  • ➦ function bar()

So the native code propagates the error back to foo, because that’s the method that called setTimeout / addEventListener, right?

  • window.onerror
  • ➦ function foo()
  • ➦ native code
  • ➦ function bar()
wrong!

The native code does not know where to put the error, since it’s a completely new execution context So the error will be delegated directly to window on error
